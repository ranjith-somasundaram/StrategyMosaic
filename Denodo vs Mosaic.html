02The core architectural difference — start here
Critical framing — before feature comparison
They are not solving the same problem
A direct feature-by-feature comparison between Denodo and Mosaic produces apparent overlap that obscures the real architectural distinction. The right question is not "which has more connectors" — it is where business definitions live and what enforces their consistency.
Denodo — what it governs
  • A KPI lives inside a SQL view as an expression — no independent existence outside that view
  • Multiple teams can define the same metric differently across different views
  • The platform has no mechanism to detect or enforce consistency between conflicting view definitions
  • Consistency requires all teams and tools to agree on which view to consume — governed by discipline outside Denodo
Denodo solves data access — live, federated, without replication.
Mosaic — what it governs
  • A KPI is a standalone certified metric object with an owner, a single definition, and lineage
  • Security policies are attached directly to the metric object — not to any individual view or tool
  • Power BI, Qlik, Excel, and AI agents via MCP all receive the same certified answer
  • No tool can override a metric definition locally; no parallel version can exist without a formal governance action
Mosaic solves business definition governance — consistent by design, not by discipline.
03Where they genuinely overlap — acknowledge this

A direct feature-by-feature comparison will produce apparent overlap. Acknowledging this builds credibility before the real differentiation is made.

≈

≈
Both enforce row-level security, RBAC, and audit logging centrally

Considerations & scope boundaries
→
No native BI layer
  • No dashboards, reports, or visualisation product
  • Requires a separate BI tool (Tableau, Power BI, Qlik) for end-user consumption
→
Implementation complexity
  • Complex to deploy in large heterogeneous environments
  • Some connectors cited as inconsistent in reliability (Gartner Peer Insights, Capterra)
  • Data virtualization expertise required — steep learning curve for non-specialists
→
Cache is disk-based, not in-memory
  • Cache stored in an external JDBC database (embedded Derby or external RDBMS)
  • Performance gain is from avoiding source re-fetch — not RAM-resident computation
  • Lakehouse Accelerator (Presto-based MPP) available within tier core allocation
→
Metric consistency is discipline-dependent
  • KPIs live inside SQL views — no independent certified existence
  • Multiple views can carry conflicting definitions; platform does not detect conflicts
  • Consistency requires governance enforced outside Denodo
→
AI modeling is assistive, not end-to-end
  • Denodo Assistant helps with join suggestions and natural language queries
  • Not designed to build semantic models from raw source data end-to-end
→
Usage-based pricing scales with AI workloads
  • Costs tied to data volume processed and data products queried
  • High-frequency AI agent queries increase billable consumption unpredictably
